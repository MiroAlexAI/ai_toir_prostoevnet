Общие данные
AI-ассистент "Инженер по надёжности и НСИ": Это интеллектуальный помощник, ориентированный на инженерию надёжности (Reliability Engineering) и нормативно-справочную информацию (НСИ, что в русском контексте часто означает "Нормативно-Справочная Информация" — система стандартов, справочников и данных для технического обслуживания). Ассистент будет анализировать оборудование, генерировать таблицы для анализа рисков и планирования обслуживания с использованием LLM (Large Language Model, большая языковая модель, такая как GPT или аналогичная, для обработки естественного языка и генерации контента).
Back-end с API LLM на Node.js: Предполагается, что серверная часть уже готова. Node.js — это среда выполнения JavaScript на сервере, позволяющая создавать масштабируемые приложения. API LLM — интерфейс прикладного программирования для взаимодействия с LLM, где запросы пользователей передаются в стороннюю модель (например, OpenAI API), и ответы возвращаются без сохранения данных.
Antigravity или Cursor: Antigravity может ссылаться на экспериментальный или шутливый инструмент (в Python есть модуль antigravity, отсылающий к xkcd-комиксу), но в контексте кодинга я интерпретирую это как "Aider" или аналогичный AI-инструмент для помощи в написании кода. Cursor — это AI-усиленный редактор кода (на базе VS Code), который использует LLM для автодополнения, рефакторинга и генерации кода. План ориентирован на использование Cursor для фронтенда и интеграции, так как он идеален для быстрой разработки UI/UX с AI-поддержкой. Если под Antigravity подразумевается другой инструмент (например, для автоматизации), план можно адаптировать.
FMEA (Failure Mode and Effects Analysis): Анализ видов и последствий отказов — метод систематического выявления потенциальных сбоев в системе, их причин, последствий и мер по минимизации рисков. Таблица FMEA обычно включает столбцы: вид отказа, причина, последствия, вероятность, серьёзность, обнаруживаемость, RPN (Risk Priority Number — приоритетный номер риска).
RCM (Reliability-Centered Maintenance): Обслуживание, ориентированное на надёжность — стратегия, фокусирующаяся на поддержании функций системы с учётом критичности. Таблица RCM включает: функции оборудования, виды отказов, стратегии обслуживания (профилактика, корректировка и т.д.).
ТОиР (Техническое Обслуживание и Ремонт): Система планового и внепланового обслуживания оборудования для обеспечения его работоспособности.
CSV (Comma-Separated Values): Формат файла для табличных данных, где значения разделены запятыми; удобен для экспорта и импорта в Excel или другие инструменты.





Реализация

Реализация disclaimer (предупреждения)

Компонент Disclaimer:
Создайте модальное окно (используйте React Modal или встроенное).
Текст: "Сервис предназначен для обучения. Не вводите личные и персональные данные или корпоративную тайну. Мы ничего не сохраняем, только передаём запрос в стороннюю LLM."
Две кнопки: "OK" и "Я все понимаю". При нажатии обеих — скрыть модал и сохранить состояние в localStorage (чтобы не показывать снова).
В Cursor: Промпт "Создай React компонент для модального предупреждения с двумя кнопками подтверждения".

Интеграция в App.js:
Покажите Disclaimer при первом запуске. Если подтверждено — перейдите к вводу оборудования.
Обоснование: Это обеспечивает compliance с privacy (защита данных), предотвращая ввод конфиденциальной информации.


Этап 3: Ввод и валидация оборудования

Компонент EquipmentInput:
Форма с полями: участок (site), тип (type), модель (model), год производства (year), производитель (manufacturer).
Кнопка "Отправить".
При submit: Сформируйте промпт для LLM: "Проверь, является ли это оборудованием: [введённые данные]. Не должно содержать лишней информации. Ответь 'valid' или 'invalid' с объяснением."
Вызов API: Если 'invalid' — показать ошибку: "Такое оборудование не найдено, укажите аналог."
Если 'valid' — сохранить данные в state (useState) и перейти к шагу 1.
В Cursor: Промпт "Создай форму ввода оборудования и валидацию через LLM API".

Обоснование: LLM используется для семантической проверки, чтобы избежать мусора или вредного ввода, повышая надёжность.

Этап 4: Генерация таблиц (шаги 1-4)

Общий компонент TableGenerator:
Используйте React Table (или Material-UI Table) для отображения.
Для каждой таблицы: Сформируйте промпт на основе оборудования, вызовите API, парсите ответ в табличный формат (JSON → таблица).
Кнопка "Сохранить в CSV": Используйте PapaParse или blob для экспорта.
После каждой таблицы: Модал "Продолжить?" с кнопками Да/Нет.

Шаг 1: Таблица частей и узлов (5 строк):
Промпт: "Для оборудования [данные] сгенерируй таблицу частей и основных узлов (5 строк): столбцы - Название, Описание, Функция."
Отобразить, спросить продолжить.

Шаг 2: Таблица FMEA (до 15 строк):
Промпт: "Сгенерируй FMEA таблицу для [оборудования] (до 15 строк): столбцы - Вид отказа, Причина, Последствия, Вероятность (1-10), Серьёзность (1-10), Обнаруживаемость (1-10), RPN."
Обоснование: RPN = Вероятность * Серьёзность * Обнаруживаемость — стандартный расчёт для приоритизации.

Шаг 3: Таблица RCM (до 10 строк):
Промпт: "Сгенерируй RCM таблицу для [оборудования] (до 10 строк): столбцы - Функция, Вид отказа, Критичность (высокая/средняя/низкая), Рекомендуемая стратегия."

Шаг 4: План ТОиР и стратегии:
Промпт: "На основе RCM для [оборудования] предложи план ТОиР и стратегии ТО в зависимости от критичности (профилактика, мониторинг и т.д.)."
Отобразить как текст или таблицу.
В Cursor: Промпт "Создай компонент для последовательной генерации таблиц с промптами к LLM".
